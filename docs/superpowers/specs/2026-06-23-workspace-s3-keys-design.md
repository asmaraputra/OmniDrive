# Spec: Workspace Scoped S3 Object Storage API Keys

This document outlines the design and implementation details for adding workspace scoping to S3 API Keys (credentials) in Omnidrive. It details the hybrid access model (allowing both global and workspace-scoped keys), backend permission validation, middleware enforcement, and the Settings dashboard UI integration.

## 1. Overview & Goals

Omnidrive exposes an S3 compatibility layer. Currently, S3 API credentials are user-wide: any credential generated by a user grants full S3 access to all workspaces of which that user is a member.

### Key Goals:
- **Hybrid Scoping**: Support both global S3 keys (access to all user's workspaces) and workspace-scoped S3 keys (restricted to a single workspace).
- **Access Control**: Allow S3 key generation for a specific workspace only if the user is a `manager` or `owner` in that workspace.
- **Middleware Scoping**: Enforce workspace scoping transparently during S3 signature verification and routing.
- **Centralized Dashboard UI**: Add an "S3 API Keys" management section inside the main Settings dashboard (`SettingsPage.tsx`) with a workspace scope selector.

---

## 2. Database Schema Updates

We will add a nullable `workspace_id` column to the `s3_credentials` table to link keys directly to a specific workspace.

### Migration SQL (`packages/worker/src/db/schema.sql`)

```sql
-- Update s3_credentials to support workspace scoping
ALTER TABLE s3_credentials ADD COLUMN workspace_id TEXT REFERENCES workspaces(id) ON DELETE CASCADE;
```

*Note: For existing credentials, `workspace_id` will default to `NULL` (treating them as global keys).*

---

## 3. Backend API & Otorisasi

The existing user-level `/api/s3-credentials` route will be extended to handle workspace-scoped key creation.

### `POST /api/s3-credentials`
- **Request Body**:
  ```json
  {
    "description": "S3 Key for Backup System",
    "workspaceId": "ws-12345" // Optional: null or omitted for Global scope
  }
  ```
- **Authorization Logic**:
  1. Retrieve current authenticated user (`userId`).
  2. If `workspaceId` is provided:
     - Query `workspace_members` to fetch the user's role:
       ```sql
       SELECT role FROM workspace_members WHERE workspace_id = ? AND user_id = ?
       ```
     - Assert that `role` is either `'manager'` or `'owner'`.
     - If not, return `403 Forbidden` (`{"error": "Unauthorized to manage S3 keys for this workspace"}`).
  3. Generate key pair (access key and secret key), encrypt secret key with `TOKEN_ENCRYPTION_KEY`.
  4. Save key with `workspace_id = workspaceId` (or `NULL` if global).
  5. Return key credentials (including raw secret key) to client.

### `GET /api/s3-credentials`
- Retrieve credentials for the current user, joining workspaces to get the workspace name.
- **SQL Query**:
  ```sql
  SELECT c.id, c.access_key_id, c.description, c.created_at, c.workspace_id, w.name as workspace_name
  FROM s3_credentials c
  LEFT JOIN workspaces w ON c.workspace_id = w.id
  WHERE c.user_id = ?
  ```

## 4. S3 Router & Middleware Scoping

We must update the S3 authentication middleware and main operations to respect the `workspace_id` scope stored on the key.

### Middleware Changes (`packages/worker/src/middleware/s3-auth.ts`)
- After finding the S3 credential:
  ```typescript
  const cred = await db.prepare('SELECT * FROM s3_credentials WHERE access_key_id = ?').bind(accessKeyId).first();
  ```
- Set `s3WorkspaceId` in Hono context:
  ```typescript
  c.set('userId', cred.user_id);
  c.set('s3WorkspaceId', cred.workspace_id || null); // New: store scoped workspace ID
  ```

### Router Enforcement (`packages/worker/src/routes/s3.ts`)

1. **List Buckets (`GET /s3/`)**:
   - Filter query to only return the scoped workspace if `s3WorkspaceId` is set:
     ```sql
     SELECT w.id, w.name, w.created_at 
     FROM workspaces w
     JOIN workspace_members wm ON w.id = wm.workspace_id
     WHERE wm.user_id = ?
       AND (? IS NULL OR w.id = ?)
     ```
   - Bind `userId`, `s3WorkspaceId`, `s3WorkspaceId`.

2. **Bucket / Object Operations (`/:bucket/*` actions)**:
   - Resolve workspace query is updated to enforce `workspace_id` equality when scoped:
     ```sql
     SELECT w.id FROM workspaces w
     JOIN workspace_members wm ON w.id = wm.workspace_id
     WHERE w.name = ? AND wm.user_id = ?
       AND (? IS NULL OR w.id = ?)
     ```
   - Bind `bucketName`, `userId`, `s3WorkspaceId`, `s3WorkspaceId`.
   - If the workspace isn't resolved (either user has no access, or key is scoped to a different workspace), return XML `NoSuchBucket` or `AccessDenied` error.

## 5. Frontend Dashboard Integration

S3 Key Management will be added directly into the Settings Page (`SettingsPage.tsx`).

### API Client Additions (`packages/web/src/lib/api.ts`)
Map the S3 Credentials endpoint and Workspaces endpoint:
- `getWorkspaces: () => request<{ workspaces: { id: string; name: string; role: string }[] }>('/api/workspaces')`
- `getS3Credentials: () => request<any[]>('/api/s3-credentials')`
- `createS3Credential: (description: string, workspaceId?: string) => request<any>('/api/s3-credentials', { method: 'POST', body: JSON.stringify({ description, workspaceId }) })`
- `deleteS3Credential: (id: string) => request<{ success: boolean }>(`/api/s3-credentials/${id}`, { method: 'DELETE' })`

### Settings UI Components (`SettingsPage.tsx`)

1. **S3 Keys List**:
   - Table of generated S3 keys showing description, Access Key ID, scope status, and a delete (revoke) button.
   - **Scope Display**: If key has a `workspace_name`, display `Workspace: <workspace_name>`. Otherwise, display `Global`.

2. **Generate Key Form**:
   - Text input for Description.
   - Dropdown selection for **Scope**:
     - Option 1: `Global (All Workspaces)`
     - Options 2+: List of user's workspaces where `role === 'manager' || role === 'owner'`.

3. **Key Output Modal**:
   - Displays newly generated Access Key and Secret Key.
   - Includes warning: *"Make sure to copy the Secret Key now. It will not be shown again."*
   - Copy-to-clipboard actions.

## 6. Testing Strategy

1. **Unit & Router Tests**:
   - Verify `POST /api/s3-credentials` fails with `403` when role is `viewer` or `editor`.
   - Verify `s3AuthMiddleware` correctly sets the `s3WorkspaceId` context variable.
   - Test `s3Router` methods with `s3WorkspaceId` constraint Mock D1 databases.
2. **End-to-End Client Check**:
   - Configure S3 client (like `rclone`) with a workspace-scoped key and verify it cannot list or read files from other workspaces.
